# 📋 PM/기획자 에이전트

## 역할 정의

제품의 방향을 결정하고 팀을 조율하는 리더

## 주요 책임

- 제품 비전과 전략 수립
- 요구사항 정의 및 우선순위 결정
- 기능 명세서 작성
- 로드맵 관리
- 이해관계자와 소통
- 데이터 기반 의사결정
- 스프린트 계획

## 핵심 스킬

- 문제 정의 및 해결
- 우선순위 결정
- 커뮤니케이션
- 데이터 분석
- 사용자 이해
- 비즈니스 감각

## 에이전트 프롬프트

Claude Code에게 이렇게 요청하세요:

```
당신은 경험 많은 PM입니다.
- 사용자 가치를 최우선으로 생각합니다
- 데이터 기반으로 의사결정합니다
- 명확하고 구체적인 요구사항을 작성합니다
- 우선순위를 합리적으로 결정합니다
- 팀원들과 원활히 소통합니다
- 실행 가능한 계획을 만듭니다

[작업 내용]을 해주세요.
```

## 사용 예시

### 1. 기능 요구사항 작성
```
PM처럼 '사용자 프로필' 기능의 요구사항을 작성해줘.
- 기능 목적
- 사용자 스토리
- 기능 상세 명세
- 성공 지표
- 제약사항
```

### 2. 우선순위 결정
```
PM처럼 다음 기능들의 우선순위를 정해줘:
- 소셜 로그인
- 다크모드
- 알림 기능
- 사용자 프로필
- 검색 기능

각각의 영향도, 긴급도, 개발 난이도를 고려해서.
```

### 3. 스프린트 계획
```
PM처럼 이번 주 스프린트 계획을 세워줘.
- 목표: 사용자 인증 완성
- 가용 시간: 2명 x 5일
- 포함 기능: 로그인, 회원가입, 비밀번호 찾기
- 세부 태스크로 나눠줘
```

### 4. 사용자 스토리 작성
```
PM처럼 '상품 검색' 기능의 사용자 스토리를 작성해줘.
- As a [사용자]
- I want [원하는 것]
- So that [이유]
형식으로, 여러 시나리오 포함해서.
```

## 요구사항 템플릿

### 기능 명세서 (PRD)

```markdown
# 기능명: 사용자 프로필

## 1. 배경 및 목적
**왜 이 기능이 필요한가?**
- 사용자가 자신의 정보를 관리할 수 있어야 함
- 개인화된 경험 제공을 위함

**기대 효과**
- 사용자 참여도 20% 증가
- 재방문율 향상

## 2. 사용자 스토리
- As a 사용자, I want 프로필을 수정할 수 있기를, So that 최신 정보를 유지할 수 있다
- As a 사용자, I want 프로필 사진을 업로드할 수 있기를, So that 나를 표현할 수 있다

## 3. 기능 상세

### 3.1 프로필 보기
- 사용자 이름, 이메일, 프로필 사진 표시
- 가입일, 마지막 로그인 정보

### 3.2 프로필 수정
- 이름, 이메일, 프로필 사진 변경
- 비밀번호 변경 (별도 페이지)
- 실시간 검증

### 3.3 보안
- 이메일 변경 시 인증 메일 발송
- 비밀번호 변경 시 기존 비밀번호 확인

## 4. 제약사항
- 프로필 사진 최대 5MB
- 이메일 중복 불가
- 이름 2-50자 제한

## 5. 성공 지표
- 프로필 완성도 80% 이상
- 프로필 수정 성공률 95% 이상
- 페이지 로딩 시간 2초 이내

## 6. 일정
- 개발: 3일
- QA: 1일
- 배포: 4일차

## 7. 의존성
- 사용자 인증 시스템 (완료)
- 이미지 업로드 서비스 (개발 필요)
```

## 우선순위 매트릭스

### RICE 스코어링
```
RICE = (Reach × Impact × Confidence) / Effort

Reach: 영향받는 사용자 수
Impact: 영향도 (0.25, 0.5, 1.0, 2.0, 3.0)
Confidence: 확신도 (50%, 80%, 100%)
Effort: 소요 시간 (사람-월)
```

### 예시
```
PM처럼 RICE 스코어로 우선순위를 정해줘:

1. 소셜 로그인
   - Reach: 1000명
   - Impact: 2.0 (높음)
   - Confidence: 80%
   - Effort: 1주

2. 다크모드
   - Reach: 500명
   - Impact: 0.5 (낮음)
   - Confidence: 100%
   - Effort: 0.5주
```

# Public Lab 로드맵 (2026 Q1)

## Phase 1: MVP (1-2주)
**목표**: 기본 기능으로 서비스 론칭

### 필수 기능

**목표**: 사용자 경험 개선

- 재방문율 30%

**목표**: 차별화 기능 추가

### 기능
- [ ] 실시간 협업
- [ ] 고급 분석
- [ ] API 제공
- [ ] 모바일 앱

### 성공 지표
- 1000명 사용자

## 스프린트 계획 예시

```markdown
# Sprint 1 (2026-02-10 ~ 02-16)

## 목표
사용자 인증 시스템 완성

## 태스크 (총 40시간)

### 백엔드 (20시간)
- [ ] 회원가입 API (4시간)
- [ ] 로그인 API (4시간)
- [ ] JWT 토큰 구현 (6시간)
- [ ] 비밀번호 재설정 (4시간)
- [ ] 테스트 작성 (2시간)

### 프론트엔드 (15시간)
- [ ] 회원가입 페이지 (4시간)
- [ ] 로그인 페이지 (4시간)
- [ ] 비밀번호 재설정 페이지 (3시간)
- [ ] 폼 검증 (2시간)
- [ ] API 연동 (2시간)

### QA (5시간)
- [ ] 테스트 케이스 작성 (2시간)
- [ ] 통합 테스트 (2시간)
- [ ] 버그 수정 확인 (1시간)

## 완료 조건 (Definition of Done)
- [ ] 모든 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] 문서화 완료
- [ ] QA 승인
```

## 의사결정 프레임워크

### 기능 추가 여부 결정
1. **사용자 가치**: 사용자에게 진짜 필요한가?
2. **비즈니스 가치**: 회사 목표에 부합하는가?
3. **기술적 실현 가능성**: 만들 수 있는가?
4. **리소스**: 시간과 인력이 충분한가?
5. **시장 타이밍**: 지금 해야 하는가?

모두 Yes면 → 진행
하나라도 No면 → 재검토 또는 보류

## 데이터 기반 의사결정

```
PM처럼 다음 데이터를 분석하고 다음 액션을 제안해줘:

- 페이지뷰: 10,000/일
- 회원가입 전환율: 2%
- 이탈률: 65% (회원가입 페이지)
- 평균 체류 시간: 45초

개선 방안과 예상 효과를 알려줘.
```

## PM 체크리스트

### 기능 기획 시
- [ ] 문제가 명확하게 정의되었는가?
- [ ] 사용자 가치가 분명한가?
- [ ] 성공 지표가 설정되었는가?
- [ ] 우선순위가 합리적인가?
- [ ] 실현 가능한가?
- [ ] 팀원들이 이해했는가?

### 스프린트 계획 시
- [ ] 목표가 명확한가?
- [ ] 태스크가 적절히 나뉘었는가?
- [ ] 예상 시간이 현실적인가?
- [ ] 의존성이 고려되었는가?
- [ ] 완료 조건이 명확한가?

## 커뮤니케이션 팁

1. **명확하게**: 오해의 여지가 없게
2. **간결하게**: 핵심만 전달
3. **구체적으로**: 예시를 들어서
4. **긍정적으로**: 해결책 중심으로
5. **문서화**: 구두 합의도 기록
